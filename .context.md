## Purpose

MongoLens is a **zero-backend Chrome DevTools Extension** that:

1. Accepts Mongoose model `.js`/`.ts` files via browser file upload
2. Parses them entirely **client-side** using a custom regex + brace-counting parser
3. Extracts: collection names, field types, ObjectId refs, indexes, and model registrations
4. Builds a `SchemaGraph` (collections + relations + dead collections)
5. Renders an interactive ER-style diagram using React Flow

**Target users:** Node.js developers using Mongoose + Express who want to visualize their DB schema without leaving Chrome DevTools.

## Key Data Types

All shared types live in `src/types/schema.types.ts`. Do not duplicate them elsewhere.

```typescript
SchemaField {
  name: string
  type: string          // "String" | "Number" | "ObjectId" | "Date" | "Boolean" | "Mixed" | etc
  ref?: string          // set when type === "ObjectId" and a ref is declared
  required?: boolean
  isArray?: boolean
  unique?: boolean
}

SchemaIndex {
  fields: Record<string, 1 | -1>
  options?: { unique?: boolean; sparse?: boolean }
}

ParsedCollection {
  name: string          // model name, e.g. "User"
  variableName: string  // schema variable name, e.g. "UserSchema"
  fields: SchemaField[]
  indexes: SchemaIndex[]
  fileName: string
}

SchemaRelation {
  from: string
  to: string
  field: string
  type: 'one-to-one' | 'one-to-many' | 'many-to-many'
}

SchemaGraph {
  collections: ParsedCollection[]
  relations: SchemaRelation[]
  deadCollections: string[]   // collection names, not full objects
}
```

## Data Flow (Critical Path)

```
User File Upload (browser FileReader)
  → files: Array<{ name: string; content: string }>
  → parseFiles(files)                           [schemaParser.ts]
    → parseMongooseFile(content, fileName)
      → findSchemaDefinitions()                 ← regex + brace counting
      → parseSchemaBody()                       ← token-by-token field parser
      → extractIndexes()                        ← regex for .index() calls
      → findModelRegistrations()                ← regex for mongoose.model()
    → returns ParsedCollection[]
  → useSchemaStore.setGraph(collections)
    → buildSchemaGraph(collections)             [relationExtractor.ts]
      → extractRelations()                      ← ref fields → SchemaRelation[]
      → findDeadCollections()                   ← unreferenced collections → string[]
    → set({ graph, isDemo: false })
  → React re-renders
    → GraphCanvas rebuilds nodes + edges
    → dagre computes layout
    → ReactFlow renders
```

## State Management

**Single store**: `src/store/useSchemaStore.ts` (Zustand)

```typescript
interface SchemaStore {
  graph: SchemaGraph          // the full parsed schema graph
  selectedCollection: string | null
  isDemo: boolean             // true = showing DEMO_GRAPH, false = user-uploaded
  sidebarOpen: boolean
  setGraph(collections: ParsedCollection[]): void
  setSelectedCollection(name: string | null): void
  setSidebarOpen(open: boolean): void
  resetToDemo(): void
}
```

**Rule:** All components read from and write to this store. No prop-drilling for graph data.

## TypeScript Configuration

`tsconfig.app.json` key settings:

| Setting | Value | Effect |
|---------|-------|--------|
| `verbatimModuleSyntax` | `true` | All type imports must use `import type { ... }` |
| `skipLibCheck` | `true` | Skip type checking of node_modules |
| `noUnusedLocals` | `false` | Relaxed — extension UI has some unused vars intentionally |
| `types` | `["vite/client", "chrome"]` | Enables Chrome Extension API types via `@types/chrome` |

### Import type convention
Because `verbatimModuleSyntax: true`, all files follow this pattern:
```typescript
import type { SchemaField, ParsedCollection } from '../types/schema.types'; // types only
import { useSchemaStore } from '../store/useSchemaStore'; // runtime values
import { create } from 'zustand'; // runtime values
```

## Parser Constraints & Limitations

The parser is intentionally lightweight. It **does not handle**:
- Mongoose `discriminators`
- Schema composition via `new Schema().add()`
- Programmatically generated schemas (e.g., schema built in a loop)
- Dynamic `ref` values (e.g., `ref: someVariable`)
- TypeScript decorators (used in NestJS/typegoose patterns)

It **does handle** all patterns listed in README.md#supported-mongoose-patterns.

## Development Conventions

### Adding a new parser pattern
1. Add the pattern to `parseSchemaBody()` or `findSchemaDefinitions()` in `schemaParser.ts`
2. Add a test case to the demo data in `useSchemaStore.ts` if applicable
3. Update `README.md#supported-mongoose-patterns`

### Adding a new page/tab
1. Create `src/panel/pages/NewPage.tsx`
2. Add entry to `TABS` array in `src/panel/App.tsx`
3. Add `{activeTab === 'newtab' && <NewPage />}` render block

### Adding a new field to `SchemaField` or other types
1. Update `src/types/schema.types.ts`
2. Update `schemaParser.ts` to extract the new field
3. Update any component that renders it (search for field usages)

### Styling conventions
- Use inline `style={{}}` objects, not CSS classes
- Colors from the palette above (never hard-coded random colors)
- Hover states via `onMouseEnter`/`onMouseLeave` handlers
- Transitions: `transition: 'all 0.15s'` or `'all 0.2s ease'`
